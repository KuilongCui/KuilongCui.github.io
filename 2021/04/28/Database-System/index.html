Database Storage解决如何存储数据库，Buffer pool解决如何在内存与磁盘移动数据的问题

Lecture #01: Database Storagee (Part I)

1 Storage

本次课程讨论的是面向磁盘的DBMS，主要存储位置的在非易失性磁盘上。

易失性设备

  • 断电后数据丢失
  • 可以通过地址进行快速地随机存取
  • 这类存储设备，在本次课程中被称为内存

非易失性设备

  • 断电后数据不丢失
  • 存取以快或页为单位
  • 这类存储设备，在本次课程中被称为硬盘

2 Disk-Oriented DBMS Overview

数据库被放到磁盘上，数据库文件中的数据以页的形式组织，第一个页被称作目录页。为了操作数据，DBMS有一个buffer pool，控制数据在硬盘与内存之间的移动。DBMS还有一个执行引擎，这个执行引擎会在buffer pool中寻找特定的页，buffer pool会把对应的页加载进内存，并返回对应指针。

3 DBMS vs. OS

DBMS的目标是处理超过内存大小的数据，这和虚拟内存十分相似，所以为什么不把任务交给OS，使用mmap把文件映射到进程内存中 ？

DBMS想要自己控制所有的东西，而OS更像是一个通用货车。

  • Flushing dirty pages to disk in the correct order.

  • Specialized prefetching.

  • Buffer replacement policy.

  • Thread/process scheduling.

4 File Storage

DBMS将数据库以文件的形式存储在磁盘上，一些使用多个文件，一些存储为单个文件。而OS对文件内容一无所知，只有DBMS知道如何解释其内容。DBMS的storage manager负责管理数据库的文件，它将文件视为页的集合，持续追踪被写进页中的数据以及还有多少空闲页。

5 Database Pages

DBMS通过固定大小的块(页)组织数据库。页中可能含有不同的数据（元组、目录…），多数系统不会混合不同类型的页。一些系统要求这些页是self-contained的，这意味着对于读取本页内容所需的内容都存储在页中。

每个页都有一个唯一的identifier。如果数据库是单个文件，那么页号可以是offset。多数DBMS有一个间接层，将页号转换成文件路径与偏移，当上层查找某个页时，storage manager会将页号转换成文件路径与偏移。

多数DBMS使用固定大小的页，以避免外部碎片，通常有三种概念的页：

  • Hardware page (usually 4 KB)
  • OS page (4 KB)
  • Database page (1-16 KB).

存储设备保证原子性地进行Hardware page size大小的写，当Database page 更大时，则需要进行额外的操作。

6 Database Heap

一个heap file是一个页的无序集合，记录被乱序存储，DBMS有两种表示形式：

  • Linked List：两个指针分别指向数据页与空闲页，顺序查找
  • Page Directory: DBMS维护一个可以追踪数据页和空闲页的目录页

7 Page Layout

每个页有一个记录头，它包含当前页的元数据：

  • Page size
  • Checksum
  • DBMS version
  • Transaction visibility
  • Self-containment. (Some systems like Oracle require this.)

两种在页中存放数据的方式：

slotted-pages：记录头追踪被使用的slot，最后一个slot的偏移以及一个用于记录每个slot位置的slot数组

log-structured：相对于记录数据，DBMS仅仅记录日志。在读取数据时，DBMS扫描日志记录并生成对应数据。写速度快，读速度慢

8 Tuple Layout

Tuple Header: Contains meta-data about the tuple

Tuple Data: Actual data for attributes.

Unique Identifier: Each tuple in the database is assigned a unique identifier

Lecture #02: Database Storagee (Part II)

1 Data Representation

Integers： navie C/C++ types

Variable Precision Numbers : navie C/C++ types

Fixed-Point Precision Numbers: 任意精度与大小，但是性能略低

Variable-Length Data：任意长度的数据，溢出页

Dates and Times：unix stamp

System Catalogs：存储关于数据库的元信息：数据库中的表，每一列的数据类型等

2 Workloads

**On-Line Transaction Processing (OLTP)**：反复快速短暂的读/写操作

**On-Line Analytical Processing (OLAP)**：每次读取大量数据用于分析，复杂操作

Hybrid Transaction + Analytical Processing：OLTP+OLAP

3 Storage Models

**N-Ary Storage Model (NSM)**：

优点：快速的插入、更新和删除；对于需要整个记录的查询友好

缺点：对于需要读取表的大量数据和表的一部分属性不友好

Decomposition Storage Model (DSM)：

优点：仅会读取查询所需要的数据；便于压缩

缺点：插入、删除和更新操作较慢

如何将列存储的输出成功拼合在一起：

  • 定长属性，变长属性补0，或者使用字典
  • 内嵌tuple id

Lecture #03: Buffer Pools

1 Introduction

DBMS的目标是尽量减少数据在内存与磁盘之间移来移去。

Spatial Control：页面被写到磁盘上的什么位置，目标是使经常一起使用的页放到一起。

Temporal Control：页面什么时候被加载和写回磁盘，目标使stall的次数尽可能少。

2 Locks vs. Latches

Locks：一个锁高层的、逻辑的，她保护数据库免受其他事务的损害。事务在执行期间将持续占有锁。锁有能力回滚。

Latches：一个闩是低层的，DBMS用来保护critical sections，操作执行期间占有闩。闩不需要回滚能力。

3 Buffer Pool

buffer pool是磁盘中的页在内存中的cache，其中的每一个单元叫做frame。

Buffer Pool Meta-data

page table是一个哈希表，它完成page id到frame id的映射。由于buffer pool中的页并不是按照顺序放置的，引入中间层加快访问。此外，关于页的元数据还有dirty-bit和pin-count。

Memory Allocation Policies

Global policies：考虑所有正在运行的事务

local policies：仅考虑单个事务的快速执行，需要支持页的共享

4 Buffer Pool Optimizations

Multiple Buffer Pools：如何确定记录在哪一个buffer pool

Objected id: 在record id中附加object id

Hashing：从page id映射到buffer pool

Pre-fetching

Scan Sharing

Buffer Pool Bypass：连续大量的读取操作，将页放进进程内存中，用完直接丢弃，避免污染buffer pool

5 OS Page Cache

磁盘操作会通过OS API完成，这导致文件系统对于读取的文件会进行缓存。DBMS会通过dirct I/O告诉操作系统不要缓存读取页，因为buffer pool想要自己管理缓存。

6 Buffer Replacement Policies

Least Recently Used (LRU)

CLOCK

LRU和CLOCK如何解决 sequential flooding？

LRU-K; localization per query; priority hint

Dirty Pages

如进行修改的页，可以直接丢弃；修改的页，需要写回磁盘。

background writing

Lecture #07: Index Concurrency Control

1 Index Concurrency Control

concurrency control是DBMS在并发操作中得到“correct”的结果。

“correct”的意义：

  • Logical Correctness：线程能够读取它应该被允许读取的值
  • Physical Correctness：数据结构正确，没有野指针

今日讨论的是Physical Correctness。

2 Locks vs. Latches

3 Latch Implementations

Latch的底层实现是CPU的原子操作compare-and-swap (CAS)，有几种方法在DBMS中实现Latch：

Blocking OS Mutex

Example：std::mutex

Advantages：使用简单

Disadvantages：慢且不可扩展

Test-and-Set Spin Latch (TAS)

Example：std::atomic

Advantages：高效

Disadvantages：扩展性差，对缓存、操作系统不友好

Reader-Writer Latches

Example：std::shared_mutex

Advantages：允许并发读

Disadvantages：starvation

4 Hash Table Latching

Page Latches

Slot Latches

5 B+Tree Latching

基本的处理过程：

  1. Get latch for the parent.
  2. Get latch for the child.
  3. Release latch for the parent if it is deemed “safe”. A “safe” node is one that will not split or merge when updated (not full-on insertion or more than half full on deletion).

Basic Latch Crabbing Protocol

查找：从根节点开始，按照上述过程在父子节点之间释放与获取锁

插入/删除：从根节点开始，流程相同，需要根据子节点是否safe来释放祖先节点的锁

读操作无需关心safe问题，锁的释放顺序不重要，但是首先释放高层的锁是一个好的选择。

Improved Lock Crabbing Protocol

查找：同Basic Latch Crabbing Protocol

插入/删除：首先按照读操作进行，在叶结点上加写锁，如果叶子节点不安全，释放所有的锁，按照Basic Latch Crabbing Protocol进行

Leaf Node Scans

死锁问题，如果不能获取到锁，立刻终止执行

Delayed parent updates

Lecture #9: Joins Algorithms

1 Joins

Operator Output

Data：直接将数据输出，后续操作方便，需要大内存

Record Ids：仅输出join key和record id，late materialization

Cost Analysis

Variables used in this lecture:

  • M pages in table R (Outer Table), m tuples total
  • N pages in table S (Inner Table), n tuples total

2 Nested Loop Join

Simple Nested Loop Join

$$
Cost: M + (m*N)
$$

Block Nested Loop Join

$$
Cost: M + (M*N)
$$

Index Nested Loop Join

$$
Cost: M + (m*C)
$$

Assume the cost of each index probe is some constant C per tuple.

3 Sort-Merge Join

当表已经排好序时，表现较好

Assume that the DBMS has B buffers to use for the algorithm
$$
Sort\ Cost\ for\ R: 2M*(1+\lceil \log_{B-1} \lceil M/B \rceil \rceil)
$$

$$
Sort\ Cost\ for\ S: 2N*(1+\lceil \log_{B-1} \lceil N/B \rceil \rceil)
$$

$$
Merge\ Cost:M+N\sim M*N
$$

$$
total\ cost:Sort+Merge
$$

4 Hash Join

Basic Hash Join

总体思路是先将驱动表基于Join的属性分成更小的块，然后再进行比较。

Build：哈希处理驱动表，Key是Join的属性

Probe：使用相同哈希方法处理被驱动表，注意考虑collision，可以使用 Bloom Filter优化

Grace Hash Join / Hybrid Hash Join

适用情况：内存不足，不能完整的放下整个驱动表的哈希表

Build：使用相同哈希函数对两个表同时哈希

Probe：在对应的槽中处理

$$
Partitioning\ Phase\ Cost:2*(M+N)
$$

$$
Probe\ Phase\ Cost:M+N
$$

$$
Total\ Cost\ :3*(M+N)
$$

Lecture #10: Query Processing I

1 Query Plan

DBMS会将一个SQL语句转换为一个SQL Plan，数据从叶结点流向根节点，根节点的输出就是整个查询的输出。

2 Processing Models

processing model定义了如何执行查询，它定义数据如何在操作符间传递等。

Iterator Model

iterator model几乎被用于所有按行记录的DBMS，能够进行流水线处理，但是一些操作需要阻塞直到取出全部数据(join，subquery，order by)

Materialization Model

materialization model是特化的iterator model，每一个操作符一次会返回所有数据，对于OLTP友好

Vectorization Model

vectorization model 每次回返回一个batch大小的数据

3 Access Methods

access method定义DBMS如何存储在表中的数据，例如Processing mode的叶结点。

Sequential Scan

对于表中的每一个页，维护一个指向最后一个已经检查过的记录的cursor，顺序扫描

优化：

Prefetching

Parallelization

Buffer Pool Bypass

Zone Map：统计信息

Late Materialization：useful in column-store systems

Heap Clustering

Index Scan

4 Modification Queries

修改数据库需要进行约束性检查并且更新索引。

对于UPDATE/DELETE，必须记录之前处理过的数据，Halloween Problem。

对于INSERT，在运算符内实例化记录或者插入从子操作符传来的任何记录

5 Expression Evaluation

DBMS使用表达式树代表一个WHERE语句，灵活但是速度较慢，一个更好的方式也许是直接进行比较。

Lecture #11: Query Execution II

1 Process Models

process model定义了对于多用户的环境，系统如何处理多个请求。DBMS由许多的worker组成，它们负责处理来自客户端的任务并返回结果。

For each query plan, the DBMS decides where, when, and how to execute it. The DBMS always knows more than the OS

Process per Worker

每一个worker由一个操作系统进程表示，其依赖于OS调度。dispater收到来自请求，创建子进程处理该连接，使用shared_memory处理共享数据，单个进程的崩溃不影响整个系统。

Process Pool

类似上一个，但是CPU缓存局部性较差。

Thread per Worker

单个进程多线程处理，DBMS处理线程间的调度，减少上下文切换的开销，不必共享内存。

2 Inter-Query Parallelism

inter-query parallelism指DBMS并发处理多个查询，增加吞吐量，减少延迟，详述见Lecture #14: Concurrency Control Theory。

3 Intra-Query parallelism

intra-query parallelism并行执行单个查询，减少查询所需的时间，类似于producer/consumer。

Intra-Operator Parallelism (Horizontal)

intra-operator parallelism中，操作被分解为独立的实例，这些实例对不同的数据子集执行相同的功能。加入了一个exchange模块，以合并子操作的输出/将单个输入划分为多个。

Inter-Operator Parallelism (Vertical)

也称pipelined parallelism，类似于流水线操作。

Bushy Parallelism

同时执行一个查询的不同部分

I/O Parallelism

多个进程/线程能够加快查询的执行速度，但是磁盘容易成为瓶颈。

Multi-Disk Parallelism

磁盘阵列，DBMS甚至无法感知

Database Partitioning

将数据库划分到不同的分区中

Logical Partitioning

vertical partitioning：按照列划分为不同的区

horizontal partitioning：按照行划分为不同的区
