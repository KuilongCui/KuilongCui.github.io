Database Storage解决如何存储数据库，Buffer pool解决如何在内存与磁盘移动数据的问题

Lecture #01: Database Storagee (Part I)

1 Storage

本次课程讨论的是面向磁盘的DBMS，主要存储位置的在非易失性磁盘上。

易失性设备

  • 断电后数据丢失
  • 可以通过地址进行快速地随机存取
  • 这类存储设备，在本次课程中被称为内存

非易失性设备

  • 断电后数据不丢失
  • 存取以快或页为单位
  • 这类存储设备，在本次课程中被称为硬盘

2 Disk-Oriented DBMS Overview

数据库被放到磁盘上，数据库文件中的数据以页的形式组织，第一个页被称作目录页。为了操作数据，DBMS有一个buffer pool，控制数据在硬盘与内存之间的移动。DBMS还有一个执行引擎，这个执行引擎会在buffer pool中寻找特定的页，buffer pool会把对应的页加载进内存，并返回对应指针。

3 DBMS vs. OS

DBMS的目标是处理超过内存大小的数据，这和虚拟内存十分相似，所以为什么不把任务交给OS，使用mmap把文件映射到进程内存中 ？

DBMS想要自己控制所有的东西，而OS更像是一个通用货车。

  • Flushing dirty pages to disk in the correct order.

  • Specialized prefetching.

  • Buffer replacement policy.

  • Thread/process scheduling.

4 File Storage

DBMS将数据库以文件的形式存储在磁盘上，一些使用多个文件，一些存储为单个文件。而OS对文件内容一无所知，只有DBMS知道如何解释其内容。DBMS的storage manager负责管理数据库的文件，它将文件视为页的集合，持续追踪被写进页中的数据以及还有多少空闲页。

5 Database Pages

DBMS通过固定大小的块(页)组织数据库。页中可能含有不同的数据（元组、目录…），多数系统不会混合不同类型的页。一些系统要求这些页是self-contained的，这意味着对于读取本页内容所需的内容都存储在页中。

每个页都有一个唯一的identifier。如果数据库是单个文件，那么页号可以是offset。多数DBMS有一个间接层，将页号转换成文件路径与偏移，当上层查找某个页时，storage manager会将页号转换成文件路径与偏移。

多数DBMS使用固定大小的页，以避免外部碎片，通常有三种概念的页：

  • Hardware page (usually 4 KB)
  • OS page (4 KB)
  • Database page (1-16 KB).

存储设备保证原子性地进行Hardware page size大小的写，当Database page 更大时，则需要进行额外的操作。

6 Database Heap

一个heap file是一个页的无序集合，记录被乱序存储，DBMS有两种表示形式：

  • Linked List：两个指针分别指向数据页与空闲页，顺序查找
  • Page Directory: DBMS维护一个可以追踪数据页和空闲页的目录页

7 Page Layout

每个页有一个记录头，它包含当前页的元数据：

  • Page size
  • Checksum
  • DBMS version
  • Transaction visibility
  • Self-containment. (Some systems like Oracle require this.)

两种在页中存放数据的方式：

slotted-pages：记录头追踪被使用的slot，最后一个slot的偏移以及一个用于记录每个slot位置的slot数组

log-structured：相对于记录数据，DBMS仅仅记录日志。在读取数据时，DBMS扫描日志记录并生成对应数据。写速度快，读速度慢

8 Tuple Layout

Tuple Header: Contains meta-data about the tuple

Tuple Data: Actual data for attributes.

Unique Identifier: Each tuple in the database is assigned a unique identifier

Lecture #03: Buffer Pools

1 Introduction

DBMS的目标是尽量减少数据在内存与磁盘之间移来移去。

Spatial Control：页面被写到磁盘上的什么位置，目标是使经常一起使用的页放到一起。

Temporal Control：页面什么时候被加载和写回磁盘，目标使stall的次数尽可能少。

2 Locks vs. Latches

Locks：一个锁高层的、逻辑的，她保护数据库免受其他事务的损害。事务在执行期间将持续占有锁。锁有能力回滚。

Latches：一个闩是低层的，DBMS用来保护critical sections，操作执行期间占有闩。闩不需要回滚能力。

3 Buffer Pool

buffer pool是磁盘中的页在内存中的cache，其中的每一个单元叫做frame。

Buffer Pool Meta-data

page table是一个哈希表，它完成page id到frame id的映射。由于buffer pool中的页并不是按照顺序放置的，引入中间层加快访问。此外，关于页的元数据还有dirty-bit和pin-count。

Memory Allocation Policies

Global policies：考虑所有正在运行的事务

local policies：仅考虑单个事务的快速执行，需要支持页的共享

4 Buffer Pool Optimizations

Multiple Buffer Pools：如何确定记录在哪一个buffer pool

Objected id: 在record id中附加object id

Hashing：从page id映射到buffer pool

Pre-fetching

Scan Sharing

Buffer Pool Bypass：连续大量的读取操作，将页放进进程内存中，用完直接丢弃，避免污染buffer pool

5 OS Page Cache

磁盘操作会通过OS API完成，这导致文件系统对于读取的文件会进行缓存。DBMS会通过dirct I/O告诉操作系统不要缓存读取页，因为buffer pool想要自己管理缓存。

6 Buffer Replacement Policies

Least Recently Used (LRU)

CLOCK

LRU和CLOCK如何解决 sequential flooding？

LRU-K; localization per query; priority hint

Dirty Pages

如进行修改的页，可以直接丢弃；修改的页，需要写回磁盘。

3 Latch Implementations

Latch的底层实现是CPU的原子操作compare-and-swap (CAS)，有几种方法在DBMS中实现Latch：

4 Hash Table Latching

5 B+Tree Latching

基本的处理过程：

  1. Get latch for the parent.
  2. Get latch for the child.
  3. Release latch for the parent if it is deemed “safe”. A “safe” node is one that will not split or merge when updated (not full-on insertion or more than half full on deletion).

Basic Latch Crabbing Protocol

查找：从根节点开始，按照上述过程在父子节点之间释放与获取锁

插入/删除：从根节点开始，流程相同，需要根据子节点是否safe来释放祖先节点的锁

读操作无需关心safe问题，锁的释放顺序不重要，但是首先释放高层的锁是一个好的选择。

Improved Lock Crabbing Protocol

查找：同Basic Latch Crabbing Protocol

插入/删除：首先按照读操作进行，在叶结点上加写锁，如果叶子节点不安全，释放所有的锁，按照Basic Latch Crabbing Protocol进行

Lecture #11: Query Execution II

1 Process Models

process model定义了对于多用户的环境，系统如何处理多个请求。DBMS由许多的worker组成，它们负责处理来自客户端的任务并返回结果。

For each query plan, the DBMS decides where, when, and how to execute it. The DBMS always knows more than the OS

Process per Worker

每一个worker由一个操作系统进程表示，其依赖于OS调度。dispater收到来自请求，创建子进程处理该连接，使用shared_memory处理共享数据，单个进程的崩溃不影响整个系统。

Process Pool

类似上一个，但是CPU缓存局部性较差。

Thread per Worker

单个进程多线程处理，DBMS处理线程间的调度，减少上下文切换的开销，不必共享内存。

2 Inter-Query Parallelism

inter-query parallelism指DBMS并发处理多个查询，增加吞吐量，减少延迟，详述见Lecture #13: Concurrency Control Theory。

3 Intra-Query parallelism

intra-query parallelism并行执行单个查询，减少查询所需的时间，类似于producer/consumer。

Intra-Operator Parallelism (Horizontal)

intra-operator parallelism中，操作被分解为独立的实例，这些实例对不同的数据子集执行相同的功能。加入了一个exchange模块，以合并子操作的输出/将单个输入划分为多个。

Inter-Operator Parallelism (Vertical)

Bushy Parallelism

同时执行一个查询的不同部分

Lecture #12: Query Planning & Optimization

1 Overview

SQL语句只会声明想要什么结果，总的优化方法大致两种：

Heuristics / Rules：将语句与模式进行匹配，重写SQL语句，只会检查目录，不会检查数据，本讲内容

Cost-based Search：使用模型评估执行查询的花费，选择花费最少的，下一讲内容

Logical Plans：与查询语句等价的关系代数表达式

Physical Plans：具体的执行策略

2 Heuristics / Rules

Relational Algebra Equivalence

两个关系代数是等价的，如果它们产生相同的结果，处理步骤：

  1. 移除不可能或不必要的谓词，以及拆分连接谓词
  2. 下方选择关系
  3. 使用JOIN代替笛卡尔积
  4. 下放投影关系，移除不必要的属性

Nested Sub-Query

嵌套子查询返回一个标量或者一组数据，处理方法：

  • 重写语句，使其去相关
  • 分解嵌套查询并将结果存储到临时表

3 Cost-based Search

Cost Estimations

查询的成本取决于几个基础指标：

  • CPU：small cost, but tough to estimate
  • Disk I/O：the number of block transfers
  • Memory：the amount of DRAM used
  • Network：the number of messages sent

为估计查询的花费，DBMS在其内部目录中维护有关表、属性和索引的内部统计信息。对于每一个关系模型R，DBMS维持以下信息：
$$
N{_R}:Number\ of\ tuples\ in\ R
$$

$$
V(A,R):Number\ of\ distinct\ values\ of\ attribute\ A
$$

SC(A,R)定义了对于属性A中单个值的记录数

Selection Statistics

选择操作的结果估计方法：

  • 数据平均分布（或等宽bucket），不同谓词独立，包容原则
  • 随机采样，在样本上估计数据量

Plan Enumeration

Single-Relation Query Plans：Sequential Scan，Index Scan，启发式优化

Multi-Relation Query Plans：System R is to only consider left-deep join trees for pipe. Again, query optimization is hard…

Lecture #13: Concurrency Control Theory

1 Transactions

A transaction is the execution of a sequence of one or more operations (e.g., SQL queries) on a shared database to perform some higher level function.

  • Atomicity: Atomicity ensures that either all actions in the transaction happen, or none happen
  • Consistency: If each transaction is consistent and the database is consistent at the beginning of the transaction, then the database is guaranteed to be consistent when the transaction completes
  • Isolation: Isolation means that when a transaction executes, it should have the illusion that it is isolated from other transactions
  • Durability: If a transaction commits, then its effects on the database should persist

2 ACID: Atomicity

要么全部执行，要么全部不执行

  • Logging：DBMS记录所有的操作，以便undo操作
  • Shadow Paging：DBMS复制被修改的页，只有在事务commit之后，被修改的页才被其他事务可见。

3 ACID: Consistency

事务的执行保证逻辑性正确

Database Consistency：数据库是现实世界的模拟，完整性约束，后提交的事务应当受到早提交事务的影响。

Transaction Consistency：从一个一致性状态转换到另一个一致性状态

4 ACID: Isolation

事务之间不可见，但是事务之间可以并发执行，交错指令

Concurrency Control

Pessimistic：DBMS认为事务会冲突，所以它不会让问题出现

Optimistic：DBMS认为事务之间的冲突较少，在事务commit时，才处理冲突

Conflict Serializability

并发执行，可能产生的冲突：

  • Read-Write Conflicts (“Unrepeatable Reads”)
  • Write-Read Conflicts (“Dirty Reads”)
  • Write-Write conflict (“Lost Updates”)

如果两个调度涉及相同事务的相同操作并且每对冲突操作在两个调度中以相同方式排序，则它们是conflict equivalent。 如果调度 S 与某个串行调度conflict equivalent，则该调度 S 是冲突可串行化的。

dependency graph中没有环，则表明当前调度是冲突可串行化的

View Serializability

除了冲突可串行化外，还可以blind writes

5 ACID: Durability

所有commit的事务做出的更改必须是持久的

Lecture #14: Two-Phase Locking

1 Transaction Locks

DBMS使用locks来控制事务的执行顺序，有两种类型的lock：

  • Shared Lock (S-LOCK)
  • Exclusive Lock (X-LOCK)

2 Two-Phase Locking

Two-Phase locking (2PL)是一种pessimistic concurrency control protocol

Phase #1– Growing：请求或许所需的锁

Phase #2– Shrinking：释放不用的锁，此阶段不能再获取锁

两阶段锁是保证冲突可串行化的充分条件，但是会 cascading aborts，产生脏读（Solution：Strong Strict 2PL），和死锁（Solution: Detection or Prevention）

Strong Strict Two-Phase Locking：只在commit的时候释放锁

3 Deadlock Handling

Approach #1: Deadlock Detection：周期性检查wait-for graph是否有环，如果有环，选择一个事务abort。

Approach #2: Deadlock Prevention：当一个事务尝试获取一个已经被占有的锁，会导致两者之一被abort。基于时间戳的协议，老的时间戳意味着更高的优先级：

  • Wait-Die (“Old Waits for Young”): If the requesting transaction has a higher priority than the holding transaction, it waits. Otherwise, it aborts
  • Wound-Wait (“Young Waits for Old”): If the requesting transaction has a higher priority than the holding transaction, the holding transaction aborts and releases the lock. Otherwise, the requesting transaction waits.

4 Lock Granularities

为避免一个事务处理大量数据，导致DBMS一次性获得大量锁，占用空间，DBMS 可以使用锁层次结构，允许事务在系统中使用更粗粒度的锁。 例如，它可以通过十亿个元组而不是十亿个单独的锁获取表上的单个锁。 当事务获取此层次结构中某个对象的锁时，它会隐式获取其所有子对象的锁。

Intention locks

允许在共享模式或独占模式下锁定更高级别的节点，而无需检查所有后代节点。

  • Intention-Shared (IS): Indicates explicit locking at a lower level with shared locks
  • Intention-Exclusive (IX): Indicates explicit locking at a lower level with exclusive or shared locks
  • Shared+Intention-Exclusive (SIX): The sub-tree rooted at that node is locked explicitly in shared mode and explicit locking is being done at a lower level with exclusive-mode locks.

Lecture #15: Timestamp Ordering Concurrency Control

1 Timestamp Ordering Concurrency Control

Timestamp ordering (T/O) 是 optimistic class of concurrency control protocols，基本思想是使用时间戳来确保事务的串行化顺序，如果TS(Ti)<TS(Tj)，那么DBMS需要确定Ti发生在Tj之前。

如何生成时间戳？

System Clock；Logical Counter；Hybrid

2 Basic Timestamp Ordering (BASIC T/O)

读写记录而不需要锁，每条记录都有一个时间戳R-TS(X)或者W-TS(X)，如果事务尝试去获取来自未来的记录，事务会被abort and restart。

Read Operations

Write Operations

Optimization: Thomas Write Rule

Basic T/O protocol（不使用Thomas Write Rule）会生成冲突可串行化的调度，不会存在死锁，但是可能starvation，会产生not recoverable的调度。

A schedule is recoverable if transactions commit only after all transactions whose changes they read, commit.

Potential Issues:

  • 拷贝数据花费
  • starvation
  • 受限于时间戳分配速度

3 Optimistic Concurrency Control (OCC)

Optimistic concurrency control (OCC)是另外一种基于时间戳的optimistic concurrency control protocol。DBMS为每一个事务创造一个private workspace，所有被读写的数据会被拷贝进私有空间，事务对于数据的修改会在这个私有空间中进行，私有空间中数据的修改对于其他事务不可知。

OCC由三步组成：

**#1 – Read Phase:**Track the read/write sets of txns and store their writes in a private workspace.

**#2 – Validation Phase:**When a txn commits, check whether it conflicts with other txns.

**#3 – Write Phase:**If validation succeeds, apply private changes to database. Otherwise abort and restart the txn.

Validation Phase

DBMS会在事务开始Validation Phase时，分配时间戳。DBMS需要保证所有的冲突s go one way (from older transactions to younger transactions)，没有进入Validation Phase的时间戳设置为无穷。

If TS(Ti) < TS(Tj ), then one of the following three conditions must hold:

  • Ti completes all three phases before Tj begins
  • Ti completes before Tj starts its Write phase, and Ti does not write to any object read by Tj
  • Ti completes its Read phase before Tj completes its Read phase, and Ti does not write to any object that is either read or written by Tj

Potential Issues:

  • High overhead for copying data locally into the transaction’s private workspace
  • Validation/Write phase bottlenecks
  • Aborts are potentially more wasteful than in other protocols because they only occur after a transaction has already executed
  • Suffers from timestamp allocation bottleneck

4 Isolation Levels

Anomalies:

  • Dirty Read: Reading uncommitted data
  • Unrepeatable Reads: Redoing a read results in a different result
  • Phantom Reads: Insertion or deletions result in different results for the same range scan queries

Isolation Levels (Strongest to Weakest):

  • SERIALIZABLE: No Phantoms, all reads repeatable, and no dirty reads
  • REPEATABLE READS: Phantoms may happen
  • READ-COMMITTED: Phantoms and unrepeatable reads may happen
  • READ-UNCOMMITTED: All anomalies may happen.

5 The Phantom Problem

事务只能对于已经存在的记录加锁

Predicate Locking

Index Locking

Lecture #16: Multi-Version Concurrency Control

1 Multi-Version Concurrency Control

多版本并发控制 (MVCC) 是一个更大的概念，而不仅仅是并发控制协议。对于MVCC，DBMS对于一个逻辑记录会维护几份物理拷贝。当一个事务写一条记录时，DBMS会创建一个新版本的记录。当一个事务读取记录时，它会读取事务开始时最新的记录。

最基本的优点：写与写不会阻塞，读与读不会阻塞，没有锁，可以支持跨越时间的查询

四个重要组成部分：

Concurrency Control Protocol：2PL，timestamp ordering，occ

Version Storage

Garbage Collection

Index Management

2 Version Storage

探讨DBMS如何存储同一份记录的多份物理存储，以及事务如何找到最新的版本。

Approach #1: Append-Only Storage：所有的记录都被存储在同一张表中，不同版本的记录之间通过链表进行连接。

Approach #2: Time-Travel Storage：单独维护一个time-travel table，存储老版本的记录。对于位于 main table的最新记录，有一条指向老记录的指针。

Approach #3: Delta Storage：和Time-Travel Storage类似，但是只存储增量，更快的写，更慢的读。

3 Garbage Collection

探讨如何回收老记录(没有事务可以看到该记录或者该记录是被abort的记录创造的)

Approach #1: Tuple-level GC

  • Background Vacuuming：单独一个线程周期性扫描表，回收老记录，可以使用bitmap优化。

  • Cooperative Cleaning：工作线程检查可回收的记录当它们遍历链表时，仅O2N有效。

Approach #2: Transaction-level GC

每一个事务记录它读写的记录，当事务完成时，查看哪些记录可以回收。

4 Index Management

主键的索引指向版本链表的首部，当事务更新索引时，可以选择删除，然后重新插入。对于二级索引：

Approach #1: Logical Pointers：

二级索引指向聚簇索引，或者指向固定地址的位置（可以根据Tuple id找到记录链表）

Approach #2: Physical Pointers：一致性问题，复杂

Lecture #17: Logging Schemes

1 Crash Recovery

恢复算法由两部分组成：

  • Actions during normal transaction processing to ensure that the DBMS can recover from a failure
  • Actions after a failure to to recover the database to a state that ensures atomicity, consistency, and durability

2 Failure Classification

Type #1: Transaction Failures

  • Logical Errors: A transaction cannot complete due to some internal error condition (e.g., integrity, constraint violation)
  • Internal State Errors: The DBMS must terminate an active transaction due to an error condition (e.g., deadlock)

Type #2: System Failures

  • Software Failure: There is a problem with the DBMS implementation (e.g., uncaught divide-by-zero exception) and the system has to halt
  • Hardware Failure: The computer hosting the DBMS crashes. We assume that non-volatile storage contents are not corrupted by system crash.

Type #3: Storage Media Failure

  • Non-Repairable Hardware Failure: A head crash or similar disk failure destroys all or parts of non-volatile storage. Destruction is assumed to be detectable

3 Buffer Pool Management Policies

steal policy指事务能否将未提交的变化写到磁盘上

force policy指DBMS在事务提交时，是否将变化写到磁盘上

最简单的方法： NO-STEAL + FORCE，DBMS无需进行undo操作，也无需进行redo操作。

4 Shadow Paging

维护数据库的一份拷贝，下面是 NO-STEAL + FORCE的例子：

  • master：仅包含已提交的事务
  • Shadow：临时的数据库保存未提交的事务

回滚和恢复容易，undo时只需移除shadow pages，不需要redo；

但是拷贝页表花费较大，提交事务成本较高。

5 Write-Ahead Logging

相比shadow paging复制整个表，其实只需要保留足够的记录即可，并且可以将随机读取转化为顺序读取，Buffer pool policy一般选择STEAL + NO-FORCE。

在页面本身被允许在非易失性存储中被覆盖之前，所有与更新页面相关的日志记录都被写入到非易失性存储中。 在所有日志记录都写入稳定存储之前，不会认为事务已提交，所以在事务提交之前，需要将日志记录首先写到非易失性存储中。

6 Logging Schemes

日志记录的形式

Physical Logging: byte-level的变化被记录，如记录在页中的位置

**Logical Logging:**记录high-level的操作，如SQL

**Physiological Logging:**混合模式

Lecture #18: Database Crash Recovery

1 WAL Records

Write-ahead log records中的每一条日志都有一个log sequence number (LSN)。想要将数据写入磁盘，需要:
$$
pageLSN_{i} <= flushedLSN
$$

2 Normal Execution

Assumption:

  • All log records fit within a single page
  • Disk writes are atomic
  • Single-versioned tuples with Strict 2PL
  • STEAL + NO-FORCE buffer management with WAL

Transaction Commit

  1. 将commit写入内存中的日志
  2. 将在commit之前的所有日志写入磁盘
  3. 当commit成功后，日志中写入TXN-END

Transaction Abort

为了进行undo操作，所有的日志加入prevLSN，这种类型被叫做compensation log record (CLR)。

  1. 将abort写入内存中的日志
  2. undo事务的修改
  3. 当undo成功后，日志中写入TXN-END

3 Checkpointing

Blocking Checkpoints

  • Halt the start of any new transactions
  • Wait until all active transactions finish executing
  • Flush dirty pages to disk

Slightly Better Blocking Checkpoints

  • Halt the start of any new transactions
  • Pause transactions while the DBMS takes the checkpoint

Fuzzy Checkpoints

Active Transaction Table (ATT): The ATT represents the state of transactions that are actively running in the DBMS. A transaction’s entry is removed after the DBMS completes the commit/abort process for that transaction. For each transaction entry, the ATT contains the following information:

  • transactionId: Unique transaction identifier
  • status: The current “mode” of the transaction (Running, Committing, Undo Candidate)
  • lastLSN: Most recent LSN written by transaction

Dirty Page Table (DPT): The DPT contains information about the pages in the buffer pool that were modified by uncommitted transactions. There is one entry per dirty page containing the recLSN (i.e., the LSN of the log record that first caused the page to be dirty).

允许事务继续运行，而不必stall，添加额外的日志记录信息：

  • <CHECKPOINT-BEGIN>: Indicates the start of the checkpoint
  • <CHECKPOINT-END>: When the checkpoint has completed. It contains the ATT + DPT

4 ARIES Recovery

Phase #1 – Analysis：Read WAL from last CHECKPOINT-END to identify dirty pages in the buffer pool and active txns at the time of the crash.

Phase #2 – Redo：Repeat all actions starting from an appropriate point in the log (even txns that will abort).

Phase #3 – Undo：Reverse the actions of txns that did not commit before the crash.
